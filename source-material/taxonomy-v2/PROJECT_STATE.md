## 1. Qué es este proyecto

Sistema formal de taxonomía de entrenamiento anaeróbico/resistencia para automatizar prescripción, registro, validación y —meta v3— generación y progresión automática de rutinas con mínimo input externo. Cuatro capas de diseño cerradas; implementación (Prescriptor UI) y generador (v3) pendientes.

## 4. Decisiones arquitectónicas vigentes (índice)

1. **Inmutabilidad declarada**: family/intent/zona se declaran, no se infieren; ajustes nunca reescriben (provenance: original → effective → executed).
2. **Declarativo-primero**: YAML + motor genérico de condiciones; hooks `fn:*` solo donde el código es inevitable.
3. **Severidades**: informational / structural_hard / viability(4 niveles); `overall_coherent=false` ⟺ hard o viability.hard_fail.
4. **Catálogo**: 2 niveles (base+variación), nodos resolubles, escaleras implícitas por dimensión, regressions/equivalencias derivadas, índices ORDINALES con anclas inmutables (R3 version bump).
5. **Ratios**: modelo de hubs (back_squat global), composición transitiva con degradación de confianza; validación RELATIVA = modo primario (templates/inicios).
6. **Autoridad (Modelo C)**: todo `suggest` en v2; promoción por regla a `auto` = hook v3. Deload siempre suggest.

   Motivo POR ARTEFACTO (no homogéneo): `prescriptor.html` y `COVERAGE.md` = no-determinismo benigno / derivados regenerables (su checksum externo no aporta garantía). `compiled_rules.yaml` = determinista, pero su integridad ya la cubre su `source_digest` interno + el guard E4 del compilador (que falla si las fuentes no casan con el digest), NO su checksum en MANIFEST. En Git los 3 se siguen versionando normalmente (historial del usuario); es el MANIFEST y el chequeo de integridad los que los ignoran.
